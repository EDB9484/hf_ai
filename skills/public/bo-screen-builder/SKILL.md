# BO Screen Builder

## Overview

Spring Boot + Thymeleaf + Service 샘플 데이터 구조로 BO 화면을 최소 실행 단위로 구현한다.
요청 범위를 벗어나는 비즈니스 로직 과구현을 피하고, 재현 가능한 GET 검색 URL과 안정적인 페이징 동작을 우선한다.

## Workflow

1. 요청에서 도메인과 화면 유형을 식별한다.
2. 패키지를 `com.boauto.backoffice.admin.<domain>` 하위로 배치한다.
3. Controller에서 요청 파라미터 바인딩과 모델 조합만 담당한다.
4. Service에서 샘플 데이터, 검색/필터, 정렬, 페이징을 처리한다.
5. Thymeleaf 템플릿에 검색 폼, 결과 테이블, 페이징, 빈 결과 메시지를 구현한다.
6. 쿼리 상태를 `th:value`, `th:checked`, `th:selected`로 유지한다.
7. 변경 후 `./gradlew test`로 검증한다.

## Implementation Rules (Backend/Code)

- 도메인별로 파일을 분리하고, 관련 없는 기존 코드 리팩터링을 피한다.
- 초기 단계에서는 인메모리 샘플 데이터 기반으로 동작 가능성을 먼저 보장한다.
- Controller는 오케스트레이션만 수행하고 데이터 가공은 Service로 이동한다.
- 검색 URL은 GET 기반으로 유지해 북마크/공유 가능한 상태를 보장한다.

## UI Composition Rules (Screen/Layout)

- 검색 조건의 행 구성은 기획서 흐름을 우선하고, 임의로 한 줄에 3개씩 고정 배치하지 않는다.
- 검색 조건에 일자/일시가 있으면 텍스트 입력 대신 달력 선택(`input type="date"` 또는 `datetime-local`)을 우선 사용한다.
- 검색 버튼과 초기화 버튼은 가운데 정렬하고, 검색 버튼을 왼쪽·초기화 버튼을 오른쪽 순서로 배치한다.
- 검색 조건 영역과 버튼 영역 사이의 상하 간격은 `8px`로 고정한다.
- 검색 영역의 라벨 width를 고정하고 우측 정렬해 input 시작 위치를 수직으로 맞춘다.
- 페이지당 노출 수 선택은 검색 영역이 아니라 결과 표 우측 상단에 배치한다.
- 페이지당 노출 수 옵션은 `10, 25, 50, 100`으로 고정한다.
- 빈 결과일 때 사용자 안내 메시지를 명시적으로 렌더링한다.
- 모바일/데스크톱 최소 반응형을 제공한다.

## Interaction Rules (Behavior)

- 검색 조건에 일자/일시가 있으면 기본값을 당일로 설정한다.
- 검색 조건에 일자 범위 `전체` 버튼이 있으면 기본 범위를 `2000-01-01 ~ 2999-12-31`로 고정한다.

## Output Contract

요청이 "화면 생성"이면 기본적으로 아래 산출물을 만든다.

- Controller 1개: 라우팅 + 파라미터 바인딩 + 모델 주입
- Service 1개: 샘플 데이터 + 필터링 + 페이징
- Template 1개: 검색 폼 + 테이블 + 페이징 + 빈 결과 처리

요청이 "인터랙티브"를 포함하면 아래를 반드시 포함한다.

- 검색 파라미터의 GET 바인딩
- 필터링 반영 결과와 총 건수 표시
- 첫 페이지/마지막 페이지/빈 결과 케이스 처리
